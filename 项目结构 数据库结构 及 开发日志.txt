----------==========数据库结构==========----------
1. 用户表（users）
   - 用来存每个注册用户的基本信息
   - 字段：
     * id：用户编号，自动增加，主键
     * phone：手机号，不能重复，用于登录
     * password：登录密码（已加密）
     * nickname：昵称，可选
     * avatar：头像图片的url
     * has_seen_intro：是否看过新手引导（0=没看过，1=看过）
     * can_login：是否允许登录，TINYINT(1)，默认值为 1（允许）。用于管理员后台控制用户登录权限（例如禁用违规账号）。
     * invite_code：注册时使用的邀请码
     * created_at：注册时间
     * updated_at：信息最后更新时间

2. 用户状态表（user_status）
   - 每个用户有一条记录，记录五个方面的活力指标（0-100分）
   - 字段：
     * user_id：用户编号（关联用户表）
     * physical_vitality：身心活力值，默认50
     * emotional_tone：情绪基调值，默认50
     * relationship_connection：关系联结值，默认50
     * self_worth：自我价值值，默认50
     * meaning_direction：意义方向值，默认50
     * psychological_harmony_index：心理和谐指数，TINYINT UNSIGNED，取值范围 1-100，默认值 50。代表用户在心理层面的整体和谐程度，与其他五个维度共同构成完整的心理健康画像。
     * created_at：记录创建时间
     * updated_at：记录更新时间

3. 重大事件表（event）
   - 用户记录自己经历的重要事情
   - 字段：
     * id：事件编号，自动增加
     * user_id：哪个用户的事件
     * event_summary：事件一句话概述
     * initial_evaluation：用户对事件的初步评价（比如情绪或看法）
     * created_at：事件记录时间
     * updated_at：最后修改时间

4. 邀请码表（invite_code）
   - 存放系统生成的、有时效的邀请码，供新用户注册时使用
   - 字段：
     * id：邀请码编号
     * code：邀请码字符串（8位）
     * expiry_time：过期时间，超过这个时间就不能用了

5. 对话历史表（conversation_history）
   - 记录用户和AI助手之间的聊天内容
   - 字段：
     * id：对话记录编号
     * user_id：用户编号
     * role：消息是谁发的（'user'=用户，'assistant'=AI）
     * content：消息的具体文字内容
     * metadata：附加的JSON数据，比如AI给出的状态变更建议等
     * created_at：消息发送时间

6. 评论表（comment）
   - 很简单的表，只存评论内容、IP地址和时间
   - 字段：
     * id：评论编号
     * content：评论内容
     * ip_address：评论者IP地址
     * replied：是否已回复，TINYINT(1)，默认值为 0（未回复）。管理员可通过后台标记某条评论是否已被阅读或回应，以便筛选待处理反馈。
     * created_at：评论时间

7. 用户状态历史表（user_status_history）
   - 记录用户每次五维状态更新的完整快照，便于查看状态变化轨迹
   - 字段：
     * id：历史记录编号，自动增加，主键
     * user_id：用户编号（关联用户表）
     * physical_vitality：身心活力值（0-100）
     * emotional_tone：情绪基调值（0-100）
     * relationship_connection：关系联结值（0-100）
     * self_worth：自我价值值（0-100）
     * meaning_direction：意义方向值（0-100）
     * psychological_harmony_index：与 user_status 表保持一致，记录每次状态快照时的心理和谐指数。
     * recorded_at：状态实际更新的时间点
     * created_at：这条历史记录的入库时间

8. 系统配置表（system_config）
   - 存储全局开关、功能灰度、环境参数等键值对配置
   - 字段：
     * id：配置项编号，自动增加，主键
     * config_key：配置键名（唯一），例如 admin_login_enabled
     * config_value：配置值（文本类型，支持布尔、字符串、数字等）
     * description：配置项说明文字
     * created_at：创建时间
     * updated_at：最后更新时间
   - 初始化数据：插入一条记录，config_key='admin_login_enabled'，config_value='true'，控制管理员登录功能开关

9. 管理员操作日志表（admin_logs）
   - 记录管理员所有敏感操作的审计日志，满足安全追溯需求
   - 字段：
     * id：日志编号，自动增加，主键
     * admin_phone：执行操作的管理员手机号（冗余存储，便于快速检索）
     * action_type：操作类型（如 LOGIN、UPDATE_USER、DELETE_USER、DELETE_COMMENT、BATCH_INVITE、TOGGLE_CONFIG 等）
     * target_table：操作影响的数据表名（若涉及数据修改）
     * target_id：操作影响的记录主键ID（若涉及单条记录）
     * before_snapshot：操作前数据快照（JSON格式，记录被修改前的关键字段）
     * after_snapshot：操作后数据快照（JSON格式，记录修改后的内容）
     * request_ip：操作请求的来源IP地址
     * user_agent：操作请求的浏览器/客户端标识
     * remark：备注信息（如批量生成邀请码的数量、配置变更原因等）
     * created_at：日志记录时间
   - 索引：在 admin_phone、action_type、created_at 以及 (target_table, target_id) 上建立普通索引，优化日志查询性能

- 外键关联：用户状态、事件、对话历史都关联到用户表，删除用户时这些数据也会自动删除（级联删除）。
- 邀请码表独立，不关联用户。

4.26（深夜重构）
- 针对 DeepSeek API 接入后暴露的三大核心问题进行彻底的结构性修复：角色混淆（把助手的话当成用户的话）、时间感知错误（不知道今天几号）、追问误判（productivityAI 在追问中复述 empathy 刚说的话）。

  根因分析：
    原 AI 调用方式是将所有背景信息、历史对话、规则揉成一大段文本，作为单条 `user` 消息发送。模型无法从文本结构中区分发言人，更无法获取当前时间，完全依赖提示词中的“文字标注”。当对话轮次增多，模型极易错乱。

  重构方案（从“单条大 prompt”迁移到“标准 messages 数组”）：

  1. `ai/deepseek_client.py`
     - 新增 `deepseek_chat_messages(messages, ...)` 函数，直接接收标准的 `[{"role": "system"/"user"/"assistant", "content": ...}]` 列表，不再在内部包裹单条 `user` 消息。
     - 保留原 `deepseek_chat` 作为兼容旧版，但实际业务已不再使用。
     - 将默认温度 `TEMPERATURE` 从 0.7 下调至 **0.2**，大幅提升 JSON 输出稳定性。
     - 内部实现 `_deepseek_chat_inner` 统一处理 JSON 修正重试逻辑。

  2. `ai/empathyAI.py` 完整重写
     - 移除 `prompt_packer`，新增 `build_messages(user_message, user_info)`，返回标准 messages 列表。
     - 第一条消息为 `system`，内容包含：
         * 动态注入当前时间（精确到分钟）和星期。
         * 用户五维状态、近期事件等背景信息。
         * 小元完整人设和核心规则（原有规则全部保留）。
     - 后续依次补齐正序的历史对话（`user`/`assistant` 交替）。
     - 最后追加当前用户消息作为新的 `user`。
     - `analog_ai` 改为调用 `deepseek_chat_messages(messages)`。

  3. `ai/productivityAI.py` 完整重写
     - 同样新增 `build_messages(user_message, empathy_reply, user_info)`。
     - `system` 消息包含时间注入、背景和任务说明。
     - 历史对话正序排列。
     - 追加当前用户消息（`user`）。
     - **关键改动**：再将 empathy 刚刚生成的回复以 `assistant` 角色的形式追加到 messages 末尾。这样模型在生成后续内容时，天然知道此条是“自己刚说过的话”，彻底杜绝把助手回复当成用户发言去追问的错误。
     - 原有“禁止追问内容来源于小元自己刚说的话”等硬性文字规则保留，但不再依赖文字警告来避免混淆，结构上已杜绝。

  4. `routers/chat.py` 调用流程调整
     - 先调用 `empathy_build_messages` 构建 messages → 调用 empathy → 记录用户消息。
     - 再以 empathy_reply 为参数调用 `productivity_build_messages` 构建新的 messages（包含 empathy 回复）→ 调用 productivity。
     - 拼接最终回复、处理改名、记录 AI 回复，流程不变。

  5. 副作用修复
     - 由于温度降低至 0.3，历史对话中偶尔的过于发散或“人机感重”的问题同步改善。
     - 对话历史表存储的仍然是完整的最终回复文本（含追问和改名确认），前端展示无需额外改动。

- 此次重构从通信协议层面还原了 OpenAI Chat API 的标准用法，是接入大模型后最关键的一次底层修正。至此，AI 对话体验与官网页面对齐，不再出现“人工智障”的割裂感。

  原因：bindEvents 中同时对 chatInput 绑定了 keypress 和 keydown 两个回车事件监听，keypress 未判断 Shift 修饰键，导致 Shift+Enter 也触发了 sendMessage。
  解决：移除冗余的 keypress 监听，仅保留 keydown 中的判断逻辑（!e.shiftKey），确保纯回车发送，Shift+回车正常换行。

- 修复 productivityAI 滥用 follow_up_text 导致回复割裂的问题。
  现象：情感AI已给出完整关心和建议，工作AI仍会追加一句闲聊追问（如“现在感觉好点了吗？”），导致整体回复时间感错乱、风格不统一。
  根因：prompt 中允许“轻轻追问一句”，模型容易越过边界生成不属于数据操作的内容。
  解决：收紧 follow_up_text 规则，仅允许在三种数据相关场景下生成：
    a. 改名确认；
    b. 事件记录确认；
    c. 单一维度变化≥8时的简短健康提醒（禁止展开关心或追问细节）。
  其余情况必须留空，闲聊追问由 empathyAI 自己完成。

----------==========问题与解决==========----------
问题1.1：评论为公用评论区，发表评论无需登录，如何限制其频率？
    方案一：使用 Redis 有序集合或简单的计数+过期时间
           可鉴于该项目本就不大，使用人数极少，我不太想仅为了一个功能在项目里添加新技术 pass
    方案二：在comment表添加一列存储发送请求的IP地址 每次添加数据前都先做校验
           虽然这样会增加数据库操作量，但首先，项目性质决定了其不会有太多人使用；
           其次，评论只是一个副功能，起到及时收集反馈的作用，请求量本就不会很大。adopt

问题1.2：如何获取用户真实的IP地址？
    因为以后项目会部署到云服务器，通常会用 Nginx 做反向代理。
    直接拿 request.client.host 只能得到代理服务器的 IP（比如 127.0.0.1），拿不到真实用户的 IP。
    解决方案：从 HTTP 请求头里读取 X-Forwarded-For 或 X-Real-IP，这两个字段是代理服务器转发时附带的，里面存着用户的真实 IP。
    为了兼顾本地开发和云服务器部署，我写了一个独立工具函数 get_client_ip()，放在 utills/ip_utils.py 里。
    它通过环境变量 TRUST_PROXY_HEADERS 控制是否信任代理头：
      - 开发时设为 false，直接取 request.client.host。
      - 部署时设为 true，优先从 X-Forwarded-For 取第一个 IP。

问题2：根路由返回的html文件似乎加载不了部分css与js文件
    没刷新而已 浏览器采用了老文件 Ctrl+F5解决

问题 3.1：滚动加载时重复请求第一页。
    原因：分页状态管理混乱，加载更多时未正确递增页码。
    解决：引入 `nextPage` 变量，每次加载成功后 `nextPage++`，重置时置为 1。加载更多时传入 `reset=false` 使用当前 `nextPage` 请求。
问题 3.2：评论发布成功后页面没有立即显示新评论。
    原因：评论列表的分页状态未重置，仍在原有页码基础上继续加载。
    解决：发布成功后调用 `loadComments(true)` 重置分页并重新从第一页加载，同时滚动到顶部。
问题 3.3：原生 `alert` 打断操作，样式丑陋。
    解决：实现自定义悬浮提示系统，支持队列管理（最多 3 个），渐入渐出动画，并替换所有 `alert`。

问题 4.1：登录/注册时悬浮提示不显示，页面也不跳转。
    原因：登录/注册代码写在了 IIFE 外部，无法访问 IIFE 内部的 `showToast` 函数，导致控制台报错 `showToast is not defined`，后续代码被阻塞。
    解决：将所有登录/注册逻辑移入 IIFE 内部，并在 `init()` 函数中绑定按钮事件，确保所有函数在同一作用域内。
问题 4.2：登录成功后跳转路径错误，浏览器请求 `dashboard.html` 返回 404。
    原因：使用了相对路径 `dashboard.html`，而当前页面路径为 `/`，浏览器实际请求 `http://127.0.0.1:8080/dashboard.html`，但静态文件目录为 `/HTML/Dashboard/dashboard.html`。
    解决：跳转时使用绝对路径 `/HTML/Dashboard/dashboard.html`，与 `main.py` 中挂载的静态目录一致。

问题 5.1：修改头像时，旧头像文件未被删除，导致磁盘空间浪费。
    解决：在 `change_avatar` 接口中，成功更新数据库后调用 `delete_old_avatar` 函数，判断旧 URL 是否为默认头像，若不是则删除对应文件。同时若数据库更新失败，则回滚删除刚上传的新文件。

问题 5.2：头像上传目录和默认头像 URL 硬编码在代码中，不便于部署环境切换。
    解决：将 `AVATAR_UPLOAD_DIR` 和 `DEFAULT_AVATAR_URL` 改为从环境变量读取，并在 `.env` 文件中提供配置示例，支持开发和生产环境不同设置。

问题 6：前端上传头像时后端返回 422 Unprocessable Content，导致头像更新失败。
    原因：前端 http.js 封装对所有请求统一设置了 Content-Type: application/json，且对 data 进行了 JSON.stringify。
         当 uploadAvatar 传递 FormData 对象时，JSON.stringify(formData) 得到空对象 {}，后端 FastAPI 无法解析 multipart/form-data 格式，触发 422。
    解决：修改 http.js，增加 FormData 类型检测：
          - 若 data 为 FormData 实例，则自动移除 Content-Type 头（让浏览器设置 multipart/form-data 边界），且不进行 JSON 序列化。
          - 非 FormData 请求保持原有 JSON 处理逻辑。
          - 同时确保 Authorization 头正常附加。
    测试：替换 http.js 后，上传头像正常返回 200 及新 URL，旧头像文件自动删除，个人弹窗和顶部头像同步更新。

问题 7：前端内存处理不当。
    解决：直接在index的初始化里把可能存在的token和base_info删掉。

问题8：使用传统页码分页时，在动态插入新数据的场景下（如聊天消息、评论），会导致分页偏移、数据重复或遗漏。
    原因分析：页码分页基于固定的 `OFFSET`，当新数据插入到已加载页之前时，所有数据的绝对位置后移，相同的 `OFFSET` 将指向不同的数据集合，导致用户看到重复或跳过的内容。
    解决方案：全面采用基于游标的分页策略。
      - 聊天记录：使用自增ID作为游标，按 `id DESC` 顺序分页，每次请求 `id < before_id` 的数据，新消息的插入只会影响列表底部，不影响向上追溯历史。
      - 评论列表：因存在复合排序（优先长评论，其次时间），游标扩展为排序字段的组合 `(is_long, created_at, id)`，确保分页游标能够准确锚定位置。
    实施效果：彻底消除了因数据插入导致的页面错乱问题，用户体验显著提升，尤其在滚动加载和实时交互场景下表现稳定。

问题9：token过期时间不生效，死活不过期。
    原因：过期时间的生成应带时区，否则时间戳就会不准。而老函数.utcnow()已被弃用。
    解决：询问ai得知新写法.now(timezone.utc)以解决。
    备注：这个问题从开始困扰我到了现在（4.21）才解决，一直叫ai修也修不明白，ai还是生成不了自身认知之外的东西。

问题10：管理员和用户的token字段一致会相互覆盖。
    不想解决，一改名字好多相关存于读的地方都要改，好麻烦，并且本身也不能算作问题吧，正常来讲管理员和用户也不会同时登录。

问题11：头像上传突然报 500，日志显示 `TypeError: Unable to apply constraint 'max_length' to supplied value UploadFile(...)`。
    原因：FastAPI 版本更新导致 `File(max_length=...)` 被 Pydantic 的字符串校验器拦截，`UploadFile` 无 `len()`。
    解决：去掉 `max_length` 参数，改为读取文件内容后手动检查大小，返回 413 状态码。

问题12：管理员编辑、删除数据后需要加载渲染最新数据，两个请求并行导致了异步竞态。
    原因：编辑/删除成功后，前端会同时发起一个用于更新列表的 GET 请求和下一个操作的请求（或页面自动刷新），由于网络延迟不确定，GET 请求的响应可能晚于预期，覆盖了后续操作造成的实际最新数据，导致表格显示错乱。
    尝试方案：
      - 使用请求序列号 (loadDataSeq) 拦截旧响应：能解决部分场景，但仍可能因为请求发出的时机细微差异而失效，且增加了状态管理复杂度；
      - 在操作后延迟一段时间再请求数据：体验差且不可靠。
    最终方案：放弃自动重新拉取数据，改为前端乐观更新。编辑或删除后，直接在本地修改数据源并重绘表格，没有网络请求，因此不存在竞态。同时增加手动刷新按钮作为数据校准的兜底手段。该方案在管理后台这种并发写入极少的场景下完全可用，显著简化了异步状态管理


----------==========学到的思维==========----------
1. 在设计表结构时，要多想想以后会不会改
2. 做这个项目让我第一次意识到“降级行为”
  过去只知道异常捕获，出问题了就报问题，从来没想过需要将行为或逻辑“降级”来保证用户的体验
  出问题了也要尽量让用户觉得系统还在好好工作，这算是产品思维了
3. 才知道游标分页这个东西
  原本只会传统分页 只能解决静态文件的批量加载
  游标分页可以解决在加载的同时加入新数据的问题
  但查找指定页数的数据还是只能用传统分页
4. 游标分页的优势与应用场景：
  传统页码分页（`LIMIT offset, count`）适用于数据相对静态、插入操作极少的场景（如后台管理列表），但在用户频繁交互、新数据持续产生的场景（如聊天、动态评论流）中，页码分页存在天然缺陷，极易导致数据错乱。
  游标分页通过记录当前数据集的边界值（如最后一条记录的ID或排序键），每次查询“边界之后”的数据，天然免疫数据插入带来的偏移问题，是构建稳定实时信息流的首选方案。
  实施游标分页时，必须确保游标字段能够唯一确定位置且与排序规则严格一致；对于多字段排序的列表，需要构建复合游标，并正确使用数据库的比较语法（如 SQLAlchemy 的 `tuple_`）。
  本次重构加深了对数据库分页本质的理解，并认识到根据业务特性选择合适的分页方案对用户体验的决定性影响。
5. 大框架与细节的学习策略（AI辅助时代的认知悖论）：
  技术学习可以遵循“大框架必须会，细节可以现用现学”的原则。把精力放在分辨什么值得深挖的判断力上，而非背诵具体参数。
  核心-边缘学习模型：大框架决定项目能否跑通，边缘细节决定落地体验。高特异性、低频使用的知识点（如自定义滚动条伪元素）交给AI生成，自己负责翻译描述、判别兼容性、组件化封装。
6. 认知边界的三个层次：
  - 你没法看见认知之外的东西（提问的天花板）
  - 你没法用AI生成认知之外的东西（输出的天花板）
  - 就算得到了认知之外的东西，你也无法驾驭它（控制权的天花板）
  破解方法：保持对“目录索引”的敏感度。不需要背下具体属性，但必须知道“这件事存在、归哪个大类管、有什么潜在坑”。用归类能力把未知地带变成填空题，用“接口输入输出”的思维驾驭AI生成的黑盒代码。
7. 工具与认知的共生关系：
  AI无法替你突破认知上限，但能极大加速你在已知半径内的探索效率。真正的进步发生在你拿着AI给出的答案反向查漏补缺，把“认知之外”转化为“目录索引”的过程中。保持“一边用AI省力，一边警惕AI封印认知”的清醒，比盲目背诵API更重要。
8. 面对难以根治的异步竞态，有时改变交互模式（从“自动同步”变为“手动确认+乐观更新”）比继续修补竞态逻辑更加务实。技术选型应当匹配业务的实际并发与一致性需求。
